Savers & plotters¶

Savers persist measurement data; plotters render it. Both are flat resources: unlike instruments, they have no hardware addressing and no parent/child hierarchy. Each configured instance is one YAML file identified by a user-given name, managed by flat_resource_io.py.

The flat-resource model¶

A saver/plotter is a Params/Instrument pair, just like instruments:

  • SaverParams / PlotterParams base classes (savers/base.py, plotters/base.py) carry enabled, attribute_name, an inst property, and create_inst().
  • The runtime base classes are the behavior ABCs: GenericSaver and GenericPlotter.

They're auto-discovered the same way instruments are — drop a SaverParams (or PlotterParams) subclass with a type: Literal[...] field into lib/savers/ (or lib/plotters/) and it appears in the GUI.

The Manage pages¶

/manage_savers and /manage_plotters are backed by a set of CRUD endpoints generated by _make_resource_endpoints in backend/main.py: GET /api/manage-{savers,plotters}, .../add, .../reset, .../remove, .../update. Each writes a config/<savers|plotters>/<type>_key_<name>.yml file.

When you create a measurement, its saver/plotter requirements are filled from these registries — you pick a configured instance by name.

The saver lifecycle¶

GenericSaver defines a run-oriented lifecycle:

saver.start_run(run_type="iv_curve", device="A7", cryostat="BlueFors-1")
for integration in ...:
    saver.write_measurement(counts=..., int_time=..., temperature=...,
                            metadata={"bias_current": 12.5, "trigger_level": 0.03})
saver.end_run()

The idea: one write_measurement call per integration, tagged with all the conditions under which it was taken. The data is flat-with-metadata, decoupled from how the measurement loop was nested. See Database for the full rationale and schema.

What exists today¶

Resource Type Status
DatabaseSaver database_saver Working — SQLite, full schema, one row per integration
StandInSaver — no-op (tests/scaffolding), ignore_in_cli
MplPlotter mpl_plotter Placeholder — stores last payload, prints; no rendering yet
BokehPlotter bokeh_plotter Placeholder — same
StandInPlotter — no-op (tests/scaffolding)

Only the database saver is real; plotters are scaffolding

DatabaseSaver is fully implemented. There is no file (CSV/HDF5) saver yet, and no working plotter — MplPlotter/BokehPlotter only record the last payload and print. See the Roadmap for what's planned.
