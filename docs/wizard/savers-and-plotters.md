# Savers & plotters

Savers persist measurement data; plotters render it. Both are **flat resources**:
unlike instruments, they have no hardware addressing and no parent/child
hierarchy. Each configured instance is one YAML file identified by a user-given
name, managed by
[`flat_resource_io.py`](../../lab_wizard/lib/utilities/flat_resource_io.py).

## The flat-resource model

A saver/plotter is a `Params`/`Instrument` pair, just like instruments:

- `SaverParams` / `PlotterParams` base classes
  ([`savers/base.py`](../../lab_wizard/lib/savers/base.py),
  [`plotters/base.py`](../../lab_wizard/lib/plotters/base.py)) carry `enabled`,
  `attribute_name`, an `inst` property, and `create_inst()`.
- The runtime base classes are the **behavior ABCs**:
  [`GenericSaver`](../../lab_wizard/lib/savers/saver.py) and
  [`GenericPlotter`](../../lab_wizard/lib/plotters/plotter.py).

They're [auto-discovered](../concepts/config-and-discovery.md#type-discovery) the
same way instruments are — drop a `SaverParams` (or `PlotterParams`) subclass
with a `type: Literal[...]` field into `lib/savers/` (or `lib/plotters/`) and it
appears in the GUI.

## The Manage pages

`/manage_savers` and `/manage_plotters` are backed by a set of CRUD endpoints
generated by `_make_resource_endpoints` in
[`backend/main.py`](../../lab_wizard/wizard/backend/main.py):
`GET /api/manage-{savers,plotters}`, `.../add`, `.../reset`, `.../remove`,
`.../update`. Each writes a `config/<savers|plotters>/<type>_key_<name>.yml` file.

When you [create a measurement](creating-measurements.md), its saver/plotter
requirements are filled from these registries — you pick a configured instance by
name.

## The saver lifecycle

[`GenericSaver`](../../lab_wizard/lib/savers/saver.py) defines a run-oriented
lifecycle:

```python
saver.start_run(run_type="iv_curve", device="A7", cryostat="BlueFors-1")
for integration in ...:
    saver.write_measurement(counts=..., int_time=..., temperature=...,
                            metadata={"bias_current": 12.5, "trigger_level": 0.03})
saver.end_run()
```

The idea: **one `write_measurement` call per integration**, tagged with all the
conditions under which it was taken. The data is flat-with-metadata, decoupled
from how the measurement loop was nested. See [Database](../data/database.md) for
the full rationale and schema.

## What exists today

| Resource | Type | Status |
|---|---|---|
| [`DatabaseSaver`](../../lab_wizard/lib/savers/database_saver.py) | `database_saver` | **Working** — SQLite, full schema, one row per integration |
| `StandInSaver` | — | no-op (tests/scaffolding), `ignore_in_cli` |

| `StandInPlotter` | — | no-op (tests/scaffolding) |

!!! warning "Only the database saver is real; plotters are scaffolding"
    `DatabaseSaver` is fully implemented. There is **no file (CSV/HDF5) saver**
    yet, and **no working plotter** — `MplPlotter`/`BokehPlotter` only record the
    last payload and print. See the [Roadmap](../roadmap.md) for what's planned.
